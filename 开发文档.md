# AI Agent 仿真运行平台

> **版本**: 0.3.0 | **技术栈**: Python / FastAPI / React / Docker Compose

AI Agent 仿真运行平台是一个**企业级多 Agent 仿真与推演平台**。用户通过场景文件夹定义角色、拓扑关系和技能，平台自动分配 Docker 容器，驱动 LLM 决策，实现多角色博弈推演，并在可视化地图上实时呈现。

---

## 一、核心能力

| 能力 | 描述 |
|------|------|
| **场景定义** | 场景文件夹（3 个 JSON + skills.py），定义角色、拓扑、技能 |
| **Agent 管理** | Docker 容器池 (brain/claude-code/openclaw)，动态分配 + 扩容 |
| **LLM 决策** | 每个 Agent 独立 LLM 大脑 (DeepSeek)，根据角色和目标自主决策 |
| **Agent 通信** | 消息总线 HTTP 中转，关系矩阵 = 通信矩阵（有 edge 才能通信） |
| **容器化运行** | Docker Compose 编排，3 种 Agent 后端，容器池复用 |
| **可视化** | Campus Atlas 地图 + Agent 实时位置 + 关系拓扑 + 悬停面板 + 日志面板 |
| **日志系统** | 结构化 JSON Lines 持久化，WebSocket 实时推送，支持导出 |

## 五、容器池机制

### 固定命名

docker-compose 使用 YAML anchors 定义固定容器名：

| 类型 | 容器名 |
|------|--------|
| brain | ag-b1, ag-b2, ag-b3, ag-b4 |
| claude-code | ag-c1, ag-c2, ag-c3 |
| openclaw | ag-o1, ag-o2, ag-o3 |
| 消息总线 | bus |
| 调度中心 | srv |

### 分配策略 (`container_runtime.py`)

1. 运行场景时，遍历场景 Agent，调用 `assign_agent(agent_id, role, name, extra_meta)`
2. `_get_or_create_container(backend)` 从对应类型的运行中容器里找第一个未分配的
3. 池耗尽 → 自动 `docker run` 创建新容器（命名 ag-{prefix}10+）
4. 场景结束后 `runtime.reset()` 释放占用标记，容器复用

每个场景 Agent 独占一个容器（不共享）。

## 六、Agent 注册与通信

### 注册流程

```text
场景启动 (_launch_containers)
  │
  ├─ assign_agent() → 分配容器, 获得 ca.url = "http://ag-c1:8000"
  │
  ├─ POST {ca.url}/reset → 清除上一轮残留 (brain 记忆 + inbox)
  │
  ├─ POST bus:9000/register?agent_id=ceo&url=http://ag-c1:8000&name=首席执行官
  │     └─ bus 注册表: agent_registry["ceo"] = "http://ag-c1:8000"
  │                     agent_registry["首席执行官"] = "http://ag-c1:8000"
  │
  └─ 开始仿真轮次
```

### 通信链路 (全部通过消息总线)

```text
Agent A /act
  │
  ├─ comm.send("cfo", "CFO", "ceo", "消息内容")
  │     └─ POST bus:9000/relay  {from_id:"cfo", to:"ceo", content:"..."}
  │           ├─ bus 查 agent_registry["ceo"] → "http://ag-c1:8000"
  │           └─ POST http://ag-c1:8000/message → Agent ceo inbox
  │
  ├─ comm.broadcast("ceo", "CEO", "全员通知", allowed={cto,cfo,pm})
  │     └─ POST bus:9000/relay  {from_id:"ceo", to:"broadcast", allowed:[...]}
  │           ├─ bus 遍历 agent_registry，过滤 allowed + 排除自己
  │           └─ 逐个 POST /message 到目标容器
  │
  └─ 通信权限: 关系矩阵 edge 定义可通信对象
       CEO ↔ CTO, CFO, PM, SALES_LEAD (有 edge)
       CEO → DEV_LEAD (无 edge → 拒绝: "无通信权限")
```

## 七、场景定义 (文件夹格式)

### 文件结构

```
scenes/network_collaboration/
├── meta_and_roles.json       # 角色定义 + 全局规则
├── instances_and_skills.json # 容器实例 + 技能绑定
├── network_topology.json     # 通信拓扑 (关系矩阵)
└── skills.py                 # Python 技能实现 (可选)
```

### meta_and_roles.json

```json
{
  "scenario_metadata": {
    "title": "企业协作仿真",
    "global_rules": "仿真周期90天。每轮代表3天..."
  },
  "roles": {
    "CEO": {
      "name": "首席执行官",
      "model_backbone": "claudecode",
      "identity": "公司CEO，负责整体战略和重大决策。",
      "core_goal": "90天内公司整体利润增长20%，市场份额提升5%。",
      "primary_interaction_paradigm": "INTERNAL_COLLABORATION"
    }
  }
}
```

- `model_backbone`: `claudecode` → claude-code 容器, `openclaw` → openclaw 容器
- role key 小写后作为 `agent_id` (CEO → ceo)

### network_topology.json (通信矩阵)

```json
{
  "sub_networks": [{
    "edges": [
      {"source": "CEO", "target": "CTO", "paradigm": "COLLABORATION"}
    ]
  }]
}
```

**关系矩阵 = 通信矩阵**：只有 edge 连接的 Agent 才能互发消息。双向通信。

## 八、LLM 集成

所有 Agent 使用 **DeepSeek API** (`deepseek-chat`)。

### 配置 (docker-compose.yml)

```yaml
environment:
  ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}    # DeepSeek key
  LLM_MODEL: deepseek-chat
  ANTHROPIC_BASE_URL: https://api.deepseek.com/anthropic  # DeepSeek Anthropic 兼容端点
```

### 决策流程

```
/decide (POST)
  ├─ 注入场景身份 (effective_id, core_goal, hidden_secret, action_space)
  ├─ 注入通信矩阵 (comm_matrix)
  ├─ 构建 prompt (系统提示 + 身份 + 已知 Agent + 收件箱)
  ├─ LLM 调用 → 解析 JSON Action
  └─ 返回 { type: "send_message", target: "cfo", content: "..." }

/act (POST)
  ├─ 检查通信权限 (target 是否在 allowed_targets 中)
  ├─ comm.send() / comm.broadcast() → bus relay
  └─ 记录日志 (success/failed)
```

### Agent 状态推送机制

Agent 状态不再由日志反推，而是由 `srv` 调度中心在调用容器 API 的生命周期中直接确定：

| 状态 | 含义 | 触发点 |
|------|------|--------|
| `created` | AgentRegistry 初始创建 | 场景 setup |
| `idle` | 已分配容器、可参与调度 | 容器分配/注册成功、仿真正常结束 |
| `thinking` | 正在调用 `{agent.url}/decide` | `ContainerRuntime.decide_all()` 发起请求前 |
| `decided` | `/decide` 成功返回 | 决策 API 返回后 |
| `acting` | 正在调用 `{agent.url}/act` | `ContainerRuntime.act_all()` 发起请求前 |
| `messaged` | 动作产生消息或广播 | `/act` 返回 `send_message` / `broadcast` 且 relay 成功 |
| `analyzed` | 动作完成技能/分析类处理 | `/act` 返回 `execute_skill` / `search` / `analyze` / `plan` |
| `send_failed` | 消息发送失败 | `/act` 返回 `relayed: false` |
| `error` | API 调用、容器分配或动作异常 | 请求异常、HTTP 错误、技能/relay 异常 |
| `stopped` | 用户手动停止仿真 | stop 请求结束后 |

`ContainerRuntime` 通过 `set_status_listener()` 通知 `server.py`。`server.py` 同步更新 `AgentRegistry` 并广播现有 WebSocket 消息：

```json
{"type": "agent_status", "data": [/* AgentRegistry.get_status() */]}
```

日志仍然写入 `global.jsonl`、`behavior.jsonl` 等文件，并推送 `agent_log`，但 `POST /api/logs/agent` 不再修改 Agent 状态。

## 九、仿真运行流程

### Setup → Launch 两步

```text
1. POST /api/simulations/setup  { scene: "network_collaboration" }
     └─ _setup_scene()
          ├─ 解析场景文件夹 → SceneDefinition
          ├─ 创建 Agent 对象 → AgentRegistry.register()
          ├─ 力导向布局计算坐标
          └─ 返回 Agent 列表 + 关系图

2. POST /api/simulations/launch
     └─ _launch_containers()
          ├─ assign_agent() 为每个 Agent 分配容器
          ├─ 容器状态重置 (POST /reset 清除 brain 记忆 + inbox)
          ├─ 注册到消息总线
          ├─ 构建通信矩阵 (workflow edges → comm_matrix)
          ├─ 运行 MAX_ROUNDS 轮:
          │    ├─ 注入 context (agents, tasks, comm_matrix)
          │    ├─ runtime.run_round() → decide_all() + act_all()
          │    └─ 按 API 生命周期推送 thinking/decided/acting/messaged/analyzed 等状态
          ├─ 停止条件: 超过 stalemate_threshold 轮无变化则提前结束
          └─ 返回结果 + 日志

注：仅支持场景文件夹格式（scenes/<name>/ 目录）。
```

## 十一、日志系统（v2 统一 schema）

`global.jsonl` 是唯一全局时间线。日志按 `layer` 分为 **Agent 应用层数据** 和 **Agent 网络层数据**：`application.jsonl` 保存应用层视图，`network.jsonl` 保存网络层视图；`communication.jsonl` 和 `behavior.jsonl` 保留为兼容过滤视图。

### 统一 schema 顶层字段

每条日志记录包含以下 16 个顶层字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `timestamp` | string | 北京时间 ISO 8601 毫秒，如 `"2026-06-15T10:27:52.305"` |
| `seq` | int | 会话内自增序号，从 1 开始 |
| `session_id` | string | 会话标识，如 `"基于多Agent星状协同的分布式扫雷博弈仿真_20260615_102737_102015"` |
| `level` | string | `INFO` \| `WARN` \| `ERROR` |
| `source` | string | 日志来源：`backend` \| `agent` \| `bus` \| `frontend` |
| `component` | string | 产生日志的组件：`srv` \| `bus` \| `dashboard` \| 具体场景 agent_id（如 `cmdr_01`、`soldier_01`） |
| `category` | string | 分类：`agent_application` \| `agent_network` \| `system` \| `frontend` \| `lifecycle` |
| `layer` | string | Agent 分层：`agent_application` \| `agent_network`；非 Agent 日志可为空或省略 |
| `event` | string | 事件名，如 `agent_message`、`session_start`、`llm_api_packet` |
| `actor` | object | 发起方信息，`{"id": "cmdr_01"}` |
| `target` | object | 目标方信息，`{"id": "soldier_07"}` |
| `action` | object | 动作描述，`{"name": "send_message", "status": "success"}` |
| `message` | string | 短摘要，仅供人类快速阅读，不承载完整数据 |
| `payload` | object | 业务数据：消息正文、推理、技能参数/结果、抓包摘要等 |
| `network` | object | 通信层元数据：IP、端口、协议、延迟、包大小、TCP flags |
| `trace` | object | 追踪信息：`round`、`talk`、`channel_id`、`correlation_id` |

---

### layer / category 分类与文件路由

| layer | category | 含义 | 写入文件 |
|-------|----------|------|---------|
| `agent_application` | `agent_application` | Agent 应用层数据：业务消息、决策/执行、LLM 调用元数据 | global.jsonl + **application.jsonl** |
| `agent_network` | `agent_network` | Agent 网络层数据：Docker HTTP 收发观测、tcpdump 聚合收/发包 | global.jsonl + **network.jsonl** |
| 空 | `system` | 系统事件（启动/完成/注册/错误/事件触发） | global.jsonl |
| 空 | `frontend` | 前端交互日志（WebSocket 连接、按钮点击） | global.jsonl |
| 空 | `lifecycle` | 容器/会话生命周期 | global.jsonl |

兼容视图：

| 文件 | 写入规则 | 说明 |
|------|----------|------|
| `communication.jsonl` | `event == "agent_message"` | 只保留 Agent 应用层业务通信，不再混入 Docker HTTP 流量 |
| `behavior.jsonl` | `event in ("decide", "act", "agent_action", "agent_decide")` 或旧 `category == "agent_behavior"` | 保留决策/执行行为视图 |

旧日志兼容：历史 `communication`、`agent_behavior`、`llm_api`、`network_capture` 记录不会回写，但查询时会按 `category/event` 自动推断到新的 `layer`。

### Agent 应用层数据（layer/category: `agent_application`）

#### 1. Agent 业务通信

**event: `agent_message`**（由 `message_bus.py` 转发时写入）

| 字段路径 | 类型 | 含义 |
|----------|------|------|
| `actor.id` | string | 发送者 agent_id |
| `target.id` | string | 接收者 agent_id（广播时为 `"broadcast"`） |
| `action.name` | string | `"relay"`（单播）或 `"broadcast"`（广播） |
| `action.status` | string | `"delivered(200)"` 或 `"broadcast(N)"` |
| `payload.content` | string | 消息正文 |
| `payload.reasoning` | string | 推理内容（可为空） |
| `network.src_ip` | string | 源 IP |
| `network.src_port` | int | 源端口（通常 0） |
| `network.dst_ip` | string | 目标 IP |
| `network.dst_port` | int | 目标端口（通常 0） |
| `network.protocol` | string | `"TCP/HTTP"` 或 `"HTTP/1.1"` |
| `network.latency_ms` | float | 往返延迟（毫秒） |
| `network.packet_len` | int | 总包长 = header_len + payload_len |
| `network.header_len` | int | 头部开销（固定 200） |
| `network.payload_len` | int | 有效载荷字节数 |
| `network.tcp_flags` | string | 模拟 TCP 标志，如 `"PSH,ACK"` |
| `network.channel_id` | string | 通信信道 ID |
| `network.message_type` | string | `"relay"` 或 `"broadcast"` |

#### 3. LLM 调用元数据

**event: `llm_api_call`**（brain/OpenCLAW/llm_parser 的 SDK/HTTP 调用）

该事件属于 Agent 应用层，只记录调用元数据和估算字节数，不记录 prompt/response 正文。

| 字段路径 | 类型 | 含义 |
|----------|------|------|
| `actor.id` | string | 调用 LLM 的场景 agent_id（服务端场景解析时可为空） |
| `target.provider` | string | LLM 提供商：`"deepseek"` \| `"anthropic"` \| `"openai"` |
| `target.model` | string | 模型名，如 `"deepseek-chat"` |
| `target.host` | string | API 主机，如 `"api.deepseek.com"` |
| `target.path` | string | API 路径，如 `"/v1/chat/completions"` |
| `action.name` | string | HTTP 方法，`"POST"` |
| `action.status` | string | 状态码，如 `"200"` |
| `payload.prompt_chars` | int | 提示词字符数 |
| `payload.messages_count` | int | 消息条数 |
| `payload.max_tokens` | int | 最大生成 token 数 |
| `payload.response_chars` | int | 响应字符数 |
| `payload.prompt_tokens` / `payload.input_tokens` | int | Provider 返回的输入 token |
| `payload.completion_tokens` / `payload.output_tokens` | int | Provider 返回的输出 token |
| `payload.total_tokens` | int | Provider 返回的总 token（如有） |
| `payload.prompt_cache_hit_tokens` | int | DeepSeek 等兼容接口返回的 prompt cache 命中 token |
| `payload.prompt_cache_miss_tokens` | int | DeepSeek 等兼容接口返回的 prompt cache 未命中 token |
| `payload.prompt_tokens_details.cached_tokens` | int | OpenAI prompt cache 命中 token |
| `payload.cache_read_input_tokens` | int | Anthropic 从缓存读取的输入 token |
| `payload.cache_creation_input_tokens` | int | Anthropic 本次写入缓存的输入 token |
| `payload.error_type` | string | 错误信息（出错时，不含敏感内容） |
| `payload.body_logged` | bool | 始终 `false`，不记录正文 |
| `network.latency_ms` | float | 调用耗时（毫秒） |
| `network.request_bytes` | int | 请求估算字节数 |
| `network.response_bytes` | int | 响应估算字节数 |
| `network.external` | bool | 始终 `true` |

TOKEN CACHE MONITOR 由后端按当前仿真 session 累计 `llm_api_call` / `llm_cli_call` 的 usage 字段，前端只渲染 `/api/logs/token-usage` 快照。归一化口径如下：

| Provider | Hit | Miss / uncached input | Output | Total |
|----------|-----|-----------------------|--------|-------|
| DeepSeek | `prompt_cache_hit_tokens` | `prompt_cache_miss_tokens` | `completion_tokens` | `hit + miss + output` |
| OpenAI | `prompt_tokens_details.cached_tokens` | `prompt_tokens - cached_tokens` | `completion_tokens` | `hit + miss + output` |
| Anthropic | `cache_read_input_tokens` | `cache_creation_input_tokens + input_tokens` | `output_tokens` | `hit + miss + output` |
| Fallback | `0` | `prompt_tokens` 或 `input_tokens` | `completion_tokens` 或 `output_tokens` | `hit + miss + output` |

注意：Anthropic 的 `cache_creation_input_tokens` 只表示本次写入缓存的 token，不等同于全部未命中输入；普通 `input_tokens` 也必须计入 miss/uncached。命中率只按输入维度计算：`hit / (hit + miss)`，不把 output token 混入 cache ratio。

**event: `llm_cli_call`**（Claude Code CLI 调用）

| 字段路径 | 类型 | 含义 |
|----------|------|------|
| `actor.id` | string | 调用 Claude CLI 的场景 agent_id |

| `payload.capture_source` | string | `"tcpdump"` |
| `payload.body_logged` | bool | 始终 `false`，不解密 HTTPS |
| `network.direction` | string | `"out"`（`dst_port=443/80`，发往外部 LLM）或 `"in"`（`src_port=443/80`，外部 LLM 回包） |

### 前端日志接入

前端 `logEntry()` 将日志异步上报 `POST /api/logs/ingest`，标记 `source: "frontend"`、`category: "frontend"`，进入 global.jsonl。

### API 端点

| 端点 | 用途 |
|------|------|
| `GET /api/logs` | 查询日志（支持 agent_id/level/event/layer/category/keyword 过滤；例如 `layer=agent_network`） |
| `GET /api/logs/stats` | 日志统计 |
| `GET /api/logs/agent/{id}` | 单个 Agent 时间线 |
| `GET /api/logs/messages` | Agent 应用层业务消息（只返回 `event == "agent_message"`，不包含 Docker HTTP） |
| `GET /api/logs/network` | Agent 网络层日志（Docker HTTP 收发 + tcpdump 聚合收/发包） |
| `GET /api/logs/token-usage` | 当前仿真 session 的 LLM token 累计快照（hit/miss/output/total 点序列） |
| `POST /api/logs/agent` | Agent 决策/执行日志（向后兼容，内部转为 v2 schema） |
| `POST /api/logs/ingest` | 通用日志接入（前端/LLM 抓包/流量） |
| `GET /api/logs/export?fmt=jsonl\|json\|csv` | 导出 |
| `GET /api/logs/export/file?fmt=jsonl` | 导出到文件 |
| `GET /api/logs/files` | 历史日志文件列表 |
| `GET /api/logs/download/{path}` | 下载日志文件 |
| WebSocket `agent_log` | 实时推送 Agent 行为日志 |

### 存储与导出

- 结构化日志: JSON Lines 持久化 (`agent_network/logger.py`，v2 统一 schema)
- 全局视图: `global.jsonl` 写入全部日志
- 应用层视图: `application.jsonl` 写入 `layer == "agent_application"`，包括 `agent_message`、`decide`、`act`、`llm_api_call`、`llm_cli_call`
- 网络层视图: `network.jsonl` 写入 `layer == "agent_network"`，包括 `docker_http_inbound`、`docker_http_outbound`、`llm_api_packet` 收/发包
- 兼容视图: `communication.jsonl` 只写 `agent_message`；`behavior.jsonl` 只写决策/执行行为
- 报文记录: TCP/IP 包级别 — `PacketRecorder`（`event_bus.py`，**内存存储，不写入 JSONL 文件**）
- 支持导出: JSONL / JSON / CSV
- WebSocket 实时推送
